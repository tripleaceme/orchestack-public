OpenMetadata

Column-level lineage, data catalog, glossary. Auto-ingests from Airbyte, dbt, and PostgreSQL to show how data flows end to end.

Tier

Cold. OpenMetadata is heavy (~2–3 GB RAM) and stakeholders rarely open it. Engineers open it when they need to understand lineage, OrcheStack spins it up on click and stops it when idle.

Role

OpenMetadata pulls metadata from every stage (Airbyte sources, PostgreSQL schemas, dbt models, Metabase dashboards) to produce column-level lineage graphs. It answers the question: "If I change this column, what breaks downstream?"

Configuration

  • Admin email and password: for the OpenMetadata web UI
  • JWT secret: 32+ characters, used for internal API authentication
  • Backend PostgreSQL: OrcheStack stores OpenMetadata's state in the openmetadata schema of the base PostgreSQL

When to open it

Typical triggers: before a schema change (to check downstream impact), after a pipeline failure (to see which models are affected), during an audit (to document data provenance).

Adding a database ingestion

OpenMetadata builds its catalog by connecting outward to each source database and crawling schemas, tables, and columns. Click Settings → Services → Databases → Add new service in the OM UI, pick the source type(PostgreSQL), then fill the connection form.

FieldValueWhy
Host and Port orchestack-postgres:5432 Container DNS name on the orchestack-net network. Port 5432 is the container-internal port.
Username warehouse_admin Dedicated warehouse role connection. This cannot be changed.
Password value of WAREHOUSE_DB_PASSWORD in .env Reveal it from Configuration → Credentials in the dashboard, or read it from your .env file.
Database data_warehouse The warehouse DB. Other DBs in this Postgres (openmetadata_db, airflow_db, airbyte_db, metabase_db) are each tool's internal storage — don't ingest them, they're noise.
Scheme postgresql+psycopg2 (default) OM's standard Postgres driver. Leave as-is.

Click Test Connection. You should see green on CheckAccess / GetDatabases / GetSchemas / GetTables. Save, then Run ingestion from the service page; OM populates the catalog within a few minutes.

Common mistakes

You putOM saysFix
localhost:postgres invalid literal for int() with base 10: 'postgres' OM parses the Host and Port field as host:port. postgres isn't a number. Use orchestack-postgres:5432.
localhost:5432 connection to server at "localhost" (::1), port 5432 failed: Connection refused Inside the OM ingestion container, localhost is the container itself — no Postgres there. Use the docker DNS name orchestack-postgres:5432.
localhost:5433 Same connection refused 5433 is the host-side port mapping (so you can psql -h localhost -p 5433 from your laptop). Other containers see Postgres at its internal port 5432.
orchestack_admin as username Works, but pollutes audit trail Use warehouse_admin. The role is provisioned specifically so data tools don't run as the platform admin.

Ingesting an external database (outside OrcheStack)

If your warehouse is hosted elsewhere — RDS, Cloud SQL, a colleague's VM — the same connection form takes the external host's reachable hostname or IP. OpenMetadata's ingestion container makes outbound network calls like any other container, so as long as DNS resolves and the firewall allows it, you're good.

  • Public RDS endpoint: my-warehouse.abc123.eu-west-1.rds.amazonaws.com:5432
  • Internal hostname over VPN: resolves via your host's DNS; make sure your Docker daemon's DNS is configured to see it (host network not required for outbound).
  • Connection arguments: add sslmode=require for cloud-managed databases.

Scheduling repeat ingestion

After the first ingestion succeeds, add a Metadata workflow with a cron schedule (e.g. 0 2 * * * for nightly at 02:00). The ingestion runs inside the orchestack-openmetadata-ingestion sidecar (Airflow under the hood), so it survives across OM server restarts.

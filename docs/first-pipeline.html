Step 4 — Run your first pipeline

With OrcheStack configured, drive the demo pipeline end-to-end: dbt materialises a model into the warehouse, Great Expectations validates the result, Metabase visualises it, Airflow can schedule it. This page walks through every step against the demo project that ships pre-configured — no GitHub repo required.

What ships pre-configured

OrcheStack provisions a working data-quality cycle out of the box, so you can verify the whole pipeline runs before connecting your own data:

Service Demo content
dbtA built-in project at /usr/app/dbt with one model — marts.warehouse_tables — that materialises a catalogue of every table in your warehouse, plus a not_null test on the table-name column.
Great ExpectationsA built-in context at /gx with one suite — demo_warehouse_tables — that validates an inline SQL query against pg_catalog.pg_tables: row count between 1 and 100 000, non-null table names, schema names mostly in OrcheStack's whitelist.
MetabaseNo demo dashboards — you connect Metabase to the warehouse the first time you open it (a 30-second flow described below).
AirflowNo pre-loaded DAGs — point Airflow at a Git repo via AIRFLOW_DAGS_REPO_URL in .env (or the dashboard's Edit config) and it syncs the repo's dags/ folder on every scheduler tick.

The remaining sections walk through running each tool against the demo data. Total time for a first pass is about 10 minutes.

Step 1 — Run dbt against the warehouse

From the dashboard's Services panel, click Open Terminal on the dbt tile. A web terminal opens at /usr/app/dbt with the demo project loaded and a generated profiles.yml pointing at your warehouse.

Run the model and its test in one shot:

$ dbt build

12:00:01  Found 1 model, 1 test, 0 snapshots, 0 analyses
12:00:03  Concurrency: 1 threads (target='prod')

12:00:03  1 of 2 OK created sql view model marts.warehouse_tables ...... [CREATE VIEW in 0.18s]
12:00:03  2 of 2 START test not_null_warehouse_tables_tablename ........ [RUN]
12:00:04  2 of 2 PASS not_null_warehouse_tables_tablename .............. [PASS in 0.05s]
12:00:04  Finished running 1 view model, 1 test in 0 hours 0 minutes and 0.85 seconds (0.85s).
12:00:04  Completed successfully

The view marts.warehouse_tables now exists in your warehouse — every other tool in OrcheStack can read it.

Step 2 — Browse the lineage in dbt-docs

From the dashboard, click Open Docs on the dbt tile. The generated docs site loads in a new tab. You'll see:

  • Project tree in the sidebar with the demo model.
  • Model page for warehouse_tables with its description, columns, compiled SQL, and the column-level not_null test.
  • Lineage graph showing the model and its pg_catalog.pg_tables source.

If you set DBT_REPO_URL later (dashboard → Services → dbt → Edit config), OrcheStack clones that repo on every container start, regenerates docs from the new project's dbt_project.yml, and serves them at the same Docs URL. Your dbt-docs URL is stable across project changes.

Step 3 — Validate the warehouse with Great Expectations

Click Open Data Docs on the Great Expectations tile. The demo suite has already run once on container start; you'll land on the docs index showing:

  • Validation Results tab — one entry for demo_warehouse_tables with the timestamp from container boot and a 2 of 3 expectations met (66.7%) badge.
  • Expectation Suites tab — the demo_warehouse_tables suite with its three expectations laid out, including the kwargs (min/max row counts, value sets, "mostly" thresholds).

Click into the validation result to see exactly which expectation failed and why. The demo intentionally ships with one failing expectation so you can see what a failure looks like — real validation suites surface real issues, and a 100%-passing demo would mislead you about how the tool reports problems.

To re-run the checkpoint against the latest warehouse state (e.g. after a dbt build), open the GE tile's Open Terminal action:

$ great_expectations checkpoint run demo_checkpoint
Calculating Metrics: 100%|██████████| 19/19 [00:00<00:00, 38.81it/s]
Validation failed!

Suite Name                                   Status     Expectations met
- demo_warehouse_tables                      ✖ Failed   2 of 3 (66.67 %)

Refresh the Data Docs page in your other tab — the new validation result is at the top of the index.

Step 4 — Connect Metabase + schedule with Airflow

Open Metabase from the dashboard. On first open, Metabase asks you to add a data source:

  1. Database type: PostgreSQL
  2. Host: orchestack-postgres
  3. Port: 5432
  4. Database name: data_warehouse
  5. Username: warehouse_admin
  6. Password: your WAREHOUSE_DB_PASSWORD from .env (or the dashboard's Credentials page)

Metabase scans the warehouse and finds the marts schema. Click + New question → Native query and run:

SELECT schemaname, COUNT(*) AS tables
FROM marts.warehouse_tables
GROUP BY schemaname
ORDER BY tables DESC;

You should see a row per schema, with row counts. Save it as a question, add it to a dashboard — that's the full raw → marts → dashboard path on real data, in under a minute.

To schedule the dbt build to run nightly:

  1. Create a Git repo with a dags/ folder containing a DAG that triggers dbt build (you can use the BashOperator or the official DbtCloudRunJobOperator; for OrcheStack's containerised dbt, BashOperator + docker compose exec dbt dbt build is the simplest path).
  2. From the dashboard → Services → Airflow → Edit config, paste the repo URL into AIRFLOW_DAGS_REPO_URL and pick the branch.
  3. Restart Airflow from the dashboard. On boot, Airflow clones the repo into /opt/airflow/dags/; the scheduler picks up your DAG within 30 seconds.
  4. Open Airflow from the dashboard, enable the DAG, set the schedule (e.g. 0 2 * * *), and you're done.

Using your own data? The flow above generalises trivially:

  • Add an Airbyte connector (dashboard → Services → Airbyte → Open) to ingest from your real source into the warehouse's raw schema.
  • Replace the demo dbt project by setting DBT_REPO_URL to your own project; the next dbt-docs refresh shows your models.
  • Replace the demo GE project by setting GE_REPO_URL to your own great_expectations directory; the next container restart picks it up.
  • Add Metabase questions/dashboards from the marts your real models produce.
  • Add the appropriate DAG to your DAGs repo to keep it all scheduled.

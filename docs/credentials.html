Managing credentials

How OrcheStack stores service credentials, how to rotate them safely, and what breaks if you edit them carelessly.

Where credentials are stored

Every service credential (database passwords, API keys, admin emails) lives in ./config/.env on the host. This file is chmod 600-protected and should be added to your .gitignore immediately.

The generated docker-compose.yml references credentials as ${VAR_NAME}, at container start, Docker Compose substitutes values from .env.

Database layout — which services have their own PG database

Every service that needs persistent state of its own gets a dedicated PostgreSQL database inside the shared orchestack-postgres instance. Database names follow the <service>_db suffix convention so they're easy to recognise when browsing in pgAdmin or psql \l:

Database Owner role What it holds

data_warehousewarehouse_adminYour data. Raw landings from Airbyte (raw schema), dbt marts (marts schema), anything analysts query. Operator-named (default data_warehouse; override via WAREHOUSE_DB_NAME).
airflow_dbairflow_adminAirflow metadata such as DAG runs, task instances, connections, variables. Created by airflow db migrate on first start.
airbyte_dbairbyte_adminAirbyte sync state for connector configs, sync history, schema discovery cache.
temporal_dbairbyte_adminTemporal workflow history. Airbyte's sidekick, created by the same pre-start hook.
temporal_visibility_dbairbyte_adminTemporal read-side queries (the "list workflows" API surface).
metabase_dbmetabase_adminMetabase's own state for dashboards, saved questions, drilldown settings, users. NOT the data Metabase visualises (that's in data_warehouse).

Services that don't get their own PG database:

  • dbt writes to data_warehouse directly (the operator's warehouse, under the marts schema). It has a PG role (dbt_admin) with full grants on that schema but no dedicated database.
  • pgAdmin stores its config in a local volume (no PG state).
  • MinIO is filesystem-only, buckets and objects live on a Docker volume, not in Postgres.
  • Great Expectations reads from data_warehouse to run validations but stores its own checkpoints on a filesystem volume.

Service-admin naming convention

Every service that needs an auto-provisioned admin account follows the same naming convention:

  • Usernames: <service>_admin e.g. dbt_admin, airflow_admin, minio_admin, warehouse_admin.
  • Email-shaped logins: <service>_admin@orchestack.local e.g. metabase_admin@orchestack.local, pgadmin_admin@orchestack.local.

The wizard pre-fills every admin field with the convention. Operators can override at wizard time or via the Edit Config page on the running dashboard, the convention is a defensible default, not a hardcoded rule.

Why: when you open .env or the credentials page and see DBT_DB_PASSWORD next to dbt_admin, the file reads like a sentence: "the dbt admin is dbt_admin." Audit logs, error messages, and onboarding docs all benefit from consistent naming across the catalogue.

OrcheStack's platform login (the operator's personal identity for the dashboard) is a separate concept, that's the email + password you set in the install wizard's first screen and is NOT subject to the convention.

Read-only fields

Some credentials are platform-decided, not operator-decided. The Edit Config page renders them with a "read-only" badge, operators can see the value but the input is disabled. The rule:

  • Hostnames (any _HOST) — Docker network service names that change when the base compose file changes, not when an operator wants them to.
  • Ports (any _PORT) — image-internal defaults; overriding here doesn't change what the container actually listens on.
  • Image tags (any _TAG) — bound to the installed bundle version.
  • OrcheStack DB password — rotating it would lock the platform out of its own metadata store. Use the platform backup/restore flow if you really need to rotate this one.

Everything else (passwords, repo URLs, admin emails, schemas, target database names) stays editable.

Querying each database in pgAdmin

Operators usually want to inspect their warehouse data, dbt's marts, or Airflow's run history from a SQL client. The recommended path is pgAdmin (already wired into the dashboard). Two things to know up front:

Each database needs its own connection

PostgreSQL can't do cross-database queries in a single statement — when you connect, you're connected to one database. The 3-part catalog.schema.table syntax works inside a session but only when catalog matches current_database():

-- Connected to data_warehouse
SELECT * FROM marts.warehouse_tables;            -- works
SELECT * FROM data_warehouse.marts.warehouse_tables;  -- works (same DB)
SELECT * FROM orchestack_db.platform.sessions;   -- ERROR: "permission denied for schema platform"
                                                  --        (the 3-part name doesn't cross databases)

A note about querying orchestack_db

Platform internals live in the platform schema of orchestack_db — user accounts, audit logs, session records, service pinning. This data is sensitive (it tracks every admin action across the install), so by design, no service-level role has access to it. Only the platform superuser orchestack_admin can read or write here.

If you need to inspect platform state (debugging an audit gap, recovering a deleted user, etc.), connect to orchestack_db with the orchestack_admin credentials. For day-to-day analytics work, you don't need this database open in pgAdmin at all.

Granting team members controlled access

The "one connection per database with the right service-admin role" pattern gives you a starting point for delegating access to a wider team:

  • Analysts need read access to data_warehouse. Give them warehouse_admin credentials (already has SELECT on marts + raw), OR create a per-analyst read-only role with GRANT SELECT ON ALL TABLES IN SCHEMA marts TO <analyst>.
  • Engineers debugging Airflow tasks need airflow_admin credentials for airflow_db. Their changes to Airflow connections + variables route through Airflow's own UI; psql access is for diagnosis, not configuration.
  • Platform admins need orchestack_admin — but treat that credential as "break glass in case of emergency," not a daily-use login. Most audit-log questions should be answered through the dashboard's Audit page; direct psql access is for cases where the dashboard itself is the problem.

Airbyte access (no login form)

Airbyte 0.50 OSS doesn't ship a user-login system. Anyone reaching the webapp URL is dropped straight into the default workspace. OrcheStack provides the access gate at the platform layer:

  • Subpath access through /app/airbyte goes through OrcheStack's forward-auth chain — your dashboard session cookie gates the request before it reaches Airbyte. Operators not logged into OrcheStack can't reach the URL.
  • Direct-port access at http://<host>:8001 (which the Open button uses because Airbyte's React app doesn't survive subpath routing) BYPASSES the auth chain. On a single-operator host this is acceptable; on a multi-tenant host you should firewall port 8001 externally and document the limitation for your team.

Rotating a credential

From the OrcheStack admin dashboard:

  1. Click the service tile.
  2. Click Edit config.
  3. Update the field.
  4. Click Save & apply.

After you click Save, OrcheStack runs a live connection test (for database-typed credentials only) before persisting the change. If the test fails, the new value is rejected and the field stays unchanged. Non-database credentials are saved without a test.

Container recreation

Docker Compose semantics: restart reuses the existing container (stale env vars); up with changed .env values detects a config hash diff and recreates. OrcheStack issues docker compose up -d --force-recreate <service> on save, which picks up the new values.

Data volumes are preserved across recreates so you don't lose history when rotating a password.

Production hardening

.env is the MVP approach. For production:

  • Docker secrets: works well with Swarm mode.
  • HashiCorp Vault: for dynamic credentials with TTLs.
  • SOPS + age: encrypted .env files safe to commit.

These are documented in the Advanced deployment section (roadmap), not wired into the default OrcheStack install.

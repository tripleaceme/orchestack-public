Step 3 — Configure services

Pick one tool per pipeline layer, provide credentials, and OrcheStack pulls the Docker images you chose while writing a docker-compose.yml and .env under the hood.

Pick your tools

After signing up, you land on the Service selection page. Each pipeline layer has a dropdown with a tested default and documented alternatives:

  • Ingestion: Airbyte (recommended), Custom Python, None
  • Orchestration: Apache Airflow (core, always-on)
  • Warehouse: PostgreSQL (recommended), ClickHouse, DuckDB
  • Data lake: MinIO (core)
  • Transformation: dbt Core (recommended), SQLMesh
  • Data quality: Great Expectations (recommended), Soda Core, None
  • Governance: OpenMetadata (recommended), DataHub, None
  • BI: Metabase (recommended), Apache Superset, Lightdash
  • Database admin UI: pgAdmin (recommended), Adminer, pgweb, None

Mix and match as you like, OrcheStack's modular design supports any combo. Recommended defaults are a great starting point if you're unsure.

Don't worry about making the "wrong" choice. You can change services later from the admin dashboard, and OrcheStack handles the heavy lifting of reconfiguring and restarting containers for you.

Enter credentials

For each service you enabled, OrcheStack shows a credentials form. The fields are specific to that tool. For example, OpenMetadata asks for:

  • Admin email and password
  • JWT secret (32+ characters)
  • PostgreSQL backend details (host, port, database, user, password)

OrcheStack runs a live connection test against every backend before allowing you to proceed. Invalid credentials fail fast, so no container rebuilds.

Where credentials go: OrcheStack writes them to a file at ./config/.env on the host, with chmod 600-protected. The docker-compose.yml references them via ${VAR} placeholders. Add ./config to your .gitignore before committing anything.

Pull and start

Once you click Save & deploy, OrcheStack does three things:

  1. Regenerates docker-compose.yml with just the services you picked.
  2. Pulls each service's Docker image (first time only and cached after).
  3. Runs docker compose up -d to start hot-tier services and register cold-tier services with the orchestrator.

Expect the first-time pull to take 20–35 minutes depending on your internet connection. Subsequent saves are near-instant.

Editing credentials later

Every variable the setup wizard collected, plus the OrcheStack platform DB credentials, image tags, and anything else you've manually added lives in a single source of truth: the .env file on the host. The orchestrator bind-mounts that file read-write into its own container at /etc/orchestack/.env; per-service Docker Compose snippets interpolate ${VAR} references against the same file on every docker compose up -d.

There are two equally valid ways to change a credential after the install:

Via the Credentials page

Sign in as an Admin and visit /app/credentials (or press g c). Every variable in .env appears as a row. Sensitive variables (anything matching *_PASSWORD, *_SECRET, *_KEY, *_TOKEN) are masked by default. Click Reveal on a row to see and edit one, or Reveal all to expose every sensitive value at once. Type the new value and click Save.

Saving immediately writes back to .env. Image tags (AUTH_TAG, ORCHESTRATOR_TAG, DASHBOARD_TAG) and the platform DB password (ORCHESTACK_DB_PASSWORD) are rendered read-only.

The audit log records that the variable was updated and by whom, but never the value itself, so credentials never leak through audit history.

Direct .env edit

Stop the stack, edit .env in your editor of choice, and bring the stack back up:

cd orchestack-runtime-*
nano .env             # change whatever you need
docker compose up -d  # picks up the new values on next start

This is the right tool for bulk rotations (e.g. rotating every per-service admin password in one editor session) or for variables not yet surfaced on the Credentials page. The Credentials page reads the same file, so whatever you change here appears there immediately on the next refresh.

Picking up the change in the running services

Editing .envthrough either path updates the file but does not automatically restart services that are already running. Container env vars are read at start time; running containers keep the values they were started with.

To rotate a credential and have it take effect now:

  1. Save the new value (via the Credentials page or by editing .env).
  2. Go to the Services page (g h), find the affected service, click Stop, then Start. The orchestrator runs docker compose up -d again, which re-interpolates the now-updated ${VAR} references.

Data volumes are preserved across restart, so you don't lose history when rotating a password.

Heads up for admins. If a service has active user sessions (e.g. someone is using pgAdmin), stopping it ends those sessions. The dashboard's Stop button asks for confirmation in that case. Schedule rotations during off-hours or coordinate with users beforehand.

First-run-only caveat. A small number of services consume their admin credentials only on first start, when their own internal store is empty. pgAdmin, Metabase, and Airflow all behave this way. After first start, changing PGADMIN_DEFAULT_PASSWORD in .env and restarting the container will not rotate the existing admin account; you must use the tool's own admin UI to change the password (and keep .env in sync so a future fresh install picks up the new value). The Credentials page surfaces this caveat inline next to first-run-only keys in a later polish pass.

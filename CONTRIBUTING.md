# Contributing to OrcheStack

Thanks for your interest in contributing. OrcheStack is an open-source
containerised data platform that integrates Airbyte, Airflow, dbt, Great
Expectations, Metabase, MinIO, OpenMetadata, and pgAdmin under a single
operator-facing dashboard. This guide explains how to set up a development
environment, propose changes, and get them merged.

If you're an operator looking to install OrcheStack rather than contribute,
start with the operator guide at https://orchestack.africa instead.

## Code of conduct

Be kind, be specific, and assume good faith. Disagreements over technical
direction are welcome; personal attacks are not. We follow the
[Contributor Covenant](https://www.contributor-covenant.org/) at a baseline.

## Ways to contribute

| Contribution | Where to start |
| --- | --- |
| Report a bug | Open an issue using the **Bug report** template |
| Propose a feature | Open an issue using the **Feature request** template; for substantial features, discuss the design before opening a PR |
| Fix a documented bug | Look for issues labelled `good first issue` or `help wanted` |
| Improve operator docs | The docs live under `docs/` and are generated by `_generate_docs.py` — edit the Python source, not the generated HTML |
| Add an integrated service | Open an issue first — adding a service touches the catalogue, the compose snippets, the pre-start hooks, and the dashboard; discuss the design before code |

## Setting up a development environment

You need: a recent Docker Engine (or Docker Desktop), Python 3.12 or later,
and Git. Apple silicon Macs work; Linux x86_64 works; Windows is supported
through WSL2 only.

```bash
git clone https://github.com/tripleaceme/orchestack-public.git
cd orchestack-public
cp system/docker/.env.example system/docker/.env
# Edit system/docker/.env — at minimum set ORCHESTACK_DB_PASSWORD
```

The `system/docker/.env` file holds local-only credentials for your dev
instance. It is git-ignored. Never commit credentials.

## Running the platform locally

The `Makefile` exposes the common operations:

```bash
make help            # list all available targets
make dev-up          # bring up the local stack
make dev-logs        # tail logs from every service
make dev-ps          # show container state
make dev-reset       # wipe the postgres volume and restart fresh
make dev-down        # stop the stack (preserves the postgres volume)
make verify          # smoke-check the local stack is serving
```

After `make dev-up`, the dashboard is reachable at http://localhost/app/
and the setup wizard at http://localhost/signup. First-time setup walks
you through creating an administrator account and choosing which
integrated services to deploy.

### Building images from source

The three OrcheStack-owned images (auth, orchestrator, dashboard) can be
rebuilt locally so your changes are reflected:

```bash
make image-auth
make image-orchestrator
make image-dashboard
```

These produce `*:dev` tags. The default `make dev-up` uses the published
`:latest` images from Docker Hub; to use your locally-built `:dev` images,
override the relevant `*_TAG` environment variable when bringing the stack
up, or edit `system/docker/docker-compose.yml` to pin `:dev` while you're
iterating.

## Making a change

### Branch naming

Use descriptive branch names. `fix/`, `feat/`, `docs/`, and `chore/`
prefixes are encouraged but not enforced:

```
fix/openmetadata-replica-count-on-startup
feat/airbyte-connector-template
docs/clarify-first-pipeline-walkthrough
chore/bump-traefik-to-v3.3
```

### Commit messages

The first line explains *what* changed and *why*, not *how*. Imagine
someone reading `git log` six months from now — they want to know what
problem the commit solved, not what files it touched.

```
fix(orchestrator): reset OpenMetadata password on every start

OpenMetadata's standard config variable doesn't control the password,
so without an explicit reset, the literal default "admin" is applied
on every fresh deployment. Operators couldn't sign in without first
running the password reset CLI manually.

Apply the reset idempotently in the post-start hook so the operator
never has to know about it.
```

### Tests and verification

OrcheStack uses end-to-end smoke procedures rather than a unit-test
suite. The smokes live in operator-facing form in the documentation;
the verification expectations are documented per-service. Before
submitting a PR, run the relevant smoke for the area you changed:

| Change touches… | Run the smoke for… |
| --- | --- |
| `system/orchestrator/` | The full service-lifecycle flow on every catalogue service |
| `system/dashboard/` | Each affected page; check the audit log records your actions |
| `system/docker/services/<name>.yml` | The per-service smoke at `docs/services/<name>.html` |
| `docs/` | Regenerate via `python3 _generate_docs.py` and visually inspect |

Container health checks fire automatically on every `make dev-up`; CI
will also run them on your PR.

## Submitting a pull request

1. Fork the repository to your own account.
2. Create a branch from `main` with a descriptive name.
3. Make your changes, including any necessary documentation updates.
4. Push your branch to your fork.
5. Open a pull request against `main`. Fill in the PR template completely.
6. CI will build the affected images and run smoke checks. The build must
   pass before review.
7. A maintainer will review. Expect questions; revisions are normal.
8. Once approved, a maintainer will merge.

### What to include in the PR description

- The problem you're solving (link the issue if one exists)
- A short walkthrough of your approach
- Anything you considered and rejected, and why
- How you verified the change works (manual steps, smoke procedure)

PRs without a clear "what problem does this solve" framing are hard to
review and often get bounced.

## What happens after you open a PR

1. **CI runs** (`ci.yml`) — builds all four images and runs container
   health checks. Takes about three to five minutes. CI must be green
   before review.
2. **Maintainer review** — a maintainer will read your code and ask
   questions. Aim for a response within three working days for the
   first round.
3. **Iteration** — address feedback by pushing additional commits to the
   same branch. Don't force-push during review unless asked.
4. **Approval and merge** — once a maintainer approves, they will squash-
   merge your PR into `main`. The squash keeps the main-branch history
   linear and readable.
5. **Release** — your change ships to operators when a maintainer cuts the
   next release. Merging to `main` does not automatically push new
   container images.

## Release process

Operators consume OrcheStack through three artefacts:

- Docker Hub images at `tripleaceme/orchestack-{auth,orchestrator,dashboard}`
- A runtime bundle (`orchestack-runtime.tar.gz`) attached to each GitHub
  release
- The operator documentation site at https://orchestack.africa

A release is cut by a maintainer with:

```bash
make tag-release VERSION=0.3.0
git push origin v0.3.0
```

The tag push triggers `release.yml`, which builds the four images, tags
them with both `:latest` and `:0.3.0`, pushes them to Docker Hub, builds
the runtime bundle, and attaches the bundle to a GitHub release.

Between releases, merges to `main` are integration-tested by CI but do
not ship to operators.

## Areas off-limits to community PRs

The following directories are maintainer-only and PRs touching them will
be redirected:

| Area | Why off-limits |
| --- | --- |
| `assets/css/main.css` and `assets/logos/` | Brand consistency — every change requires design review against the marketing site |
| Release-cutting workflow (`tag-release`, `release.yml`) | Single maintainer owns the release cadence |
| `LICENSE` | Licence changes require legal review |

Everything else is fair game. Welcome aboard.

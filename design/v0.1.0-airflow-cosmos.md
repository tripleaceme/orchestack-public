# Design — v0.1.0: Airflow 3 + dbt + Cosmos + composition docs

**Status**: draft awaiting maintainer sign-off
**Scope**: Minimal — the smallest change that makes OrcheStack deliver
on its modern-data-stack promise
**Estimated effort**: ~1 week

## The problem v0.1.0 solves

OrcheStack's value proposition is "modern data stack on a single host":
operators install once, then orchestrate ingestion → transformation →
analytics without writing integration glue between tools. To deliver
that, Airflow must be able to drive dbt against the operator's
warehouse with per-model task granularity. The pre-release-cut shipped
Airflow 2.10 with no dbt inside the container — the canonical pipeline
failed at the dbt step. That tag was yanked from the GitHub Releases
page and Docker Hub before any operator downloaded it. This is the
re-cut.

## Goals

- Operators can author DAGs that run dbt models with per-model task IDs
  in the Airflow UI (one failed model → one failed task, not the
  whole-DAG opaque failure of `BashOperator(dbt build)`)
- Platform stays tool-choice-neutral: no shipped DAG assumes the operator
  is using Airbyte for ingestion or Metabase for BI
- Cold-start performance unchanged from v0.1.0-pre-release; image
  layers cached so subsequent boots stay sub-15s
- Airflow on the current major version (3.x), not soon-to-be-legacy 2.x

## Non-goals

- No shipped starter DAG file in the operator bundle
- No example dbt project in the operator bundle
- No multi-arch images (linux/amd64 only; arm64 is a v0.2 nice-to-have)
- No SSO across tools, no multi-host, no DAG-versioning configuration
- No tutorial video; no recorded walkthrough

## Approach

The platform makes exactly one prescriptive choice: dbt-from-Airflow
via astronomer-cosmos. Every other tool choice (ingestion library, BI
tool, scheduling cadence, pipeline shape) is documented as a pattern
operators compose to fit their stack.

### 1. Custom Airflow image — `tripleaceme/orchestack-airflow`

New Dockerfile at `system/docker/airflow-image/Dockerfile`:

```dockerfile
FROM apache/airflow:3.2.2-python3.12   # current 3.2 patch on Docker Hub

LABEL org.opencontainers.image.title="orchestack-airflow" \
      org.opencontainers.image.description="Apache Airflow 3 with dbt-core, dbt-postgres, and astronomer-cosmos preinstalled for OrcheStack" \
      org.opencontainers.image.vendor="OrcheStack" \
      org.opencontainers.image.source="https://github.com/tripleaceme/orchestack-public" \
      org.opencontainers.image.documentation="https://orchestack.africa" \
      org.opencontainers.image.licenses="Apache-2.0"

RUN pip install --no-cache-dir \
    "dbt-core==1.10.x" \
    "dbt-postgres==1.10.x" \
    "astronomer-cosmos==1.8.x"

# Verify install — build fails if any binary is missing at the end.
RUN dbt --version && python -c "import cosmos; print(f'cosmos {cosmos.__version__}')"
```

**Versions confirmed at build time**, not at design time. Cosmos's
compatibility matrix changes month-to-month; we pin to whatever's
current-stable when the image first builds.

**Why bake everything in vs `_PIP_ADDITIONAL_REQUIREMENTS`**: pip
resolution at every container start is slow and brittle. Same pattern
as `orchestack-ge`, which already pre-installs Great Expectations at
build time.

### 2. Airflow version pin: 3.2.x

Cosmos 1.8.x supports Airflow 3.0+. dbt-core is independent of Airflow
version. Airflow 3.2 reached patch-stability in early 2026; if a
showstopper appears on 3.2 at build time, fallback is 3.1.x.

### 3. dbt project mount — shared docker volume

The Airflow container needs read access to the operator's dbt project
so Cosmos can parse `manifest.json` and generate per-model tasks.

```yaml
# system/docker/services/airflow.yml (excerpt)
volumes:
  - orchestack-dbt-project:/opt/airflow/dbt-project:ro

# system/docker/services/dbt.yml (excerpt)
volumes:
  - orchestack-dbt-project:/dbt/project   # rw; the operator edits here

volumes:
  orchestack-dbt-project:
    name: orchestack-dbt-project
    external: true
```

The orchestrator's pre-start hook creates the volume if it doesn't
exist. The first-pipeline doc tells operators to put dbt project files
here, either by cloning their repo via the dbt service's terminal or
by setting `DBT_REPO_URL` in `.env`.

### 4. Auto-create Airflow Connection for the warehouse

Cosmos's `PostgresUserPasswordProfileMapping` requires an Airflow
Connection named `orchestack_warehouse` with the warehouse credentials.
The orchestrator's post-start hook on the Airflow service creates this
connection automatically using the warehouse credentials from `.env`.

```python
# system/orchestrator/app/docker_ops.py — new function
async def _bootstrap_airflow_warehouse_connection(...):
    """Idempotently create the Airflow Connection Cosmos needs."""
    # Runs `airflow connections add orchestack_warehouse \
    #      --conn-type postgres --conn-host orchestack-postgres ...`
    # inside the running Airflow container. Idempotent — running it
    # twice silently no-ops on the second run.
```

Same principle as the OpenMetadata password reset: absorb the
upstream-default override at the platform layer so operators don't
have to learn about it.

### 6. Documentation — composition patterns, not prescribed DAGs

The platform's value pitch is "operators choose their stack." Shipping
a DAG that assumes Airbyte + dbt + Metabase contradicts that. Instead,
the docs teach two composition patterns operators combine to fit
their actual stack.

#### docs/services/airflow.html — new sections

**"Triggering an external tool's API"** — pattern: `HttpOperator` (in
Airflow 3, the modern replacement for Airflow 2's `SimpleHttpOperator`).
Concrete examples: trigger an Airbyte sync, refresh a Metabase
dashboard, kick off a Fivetran connector. Each example is ~10-15
lines of DAG. Operators copy the one matching their tool.

**"Running Python ingestion with custom dependencies"** — pattern:
`PythonVirtualenvOperator`. The canonical answer to "how do I use dlt
/ requests / pandas / any Python library without baking it into the
platform image?" Concrete examples: a dlt loader for a PostgreSQL
source, a `requests`-based API loader, a `pandas`-driven CSV ingester.
First run installs deps in an isolated venv (~30-60s), subsequent runs
reuse it.

**"Connection management"** — explains the auto-created
`orchestack_warehouse` connection from §4, plus how to add operator-
specific connections via the Airflow UI or `airflow connections add`.

#### docs/services/dbt.html — new sections

**"Running dbt from Airflow with Cosmos"** — the canonical pattern.
Complete `DbtDag` example showing project-config / profile-config /
execution-config wiring, with all paths pre-filled to match the
shared-volume mount from §3. Explains how Cosmos generates one task
per model + test, and shows what failure attribution looks like in
the UI.

**"Maintaining your dbt project"** — covers the two
project-population options: clone via the dbt service's terminal, or
set `DBT_REPO_URL` in `.env` for the dbt service to clone for you.

#### docs/first-pipeline.html — rewritten end-to-end

The current 236-line skeletal placeholder gets replaced with a
runnable walkthrough that respects operator tool choice. Structure:

```
1. Choose your ingestion pattern
   - Pattern A: API-triggered ingest (Airbyte, Fivetran, etc.)
       Show 15-line HttpOperator-based DAG snippet
   - Pattern B: Python-library ingest (dlt, requests, pandas, etc.)
       Show 15-line PythonVirtualenvOperator-based DAG snippet
2. Transform with dbt + Cosmos
   Show DbtDag snippet that runs after either ingestion pattern
3. Optionally refresh your BI tool
   - If Metabase: HttpOperator refresh
   - If Tableau/Superset/external: the operator's tool's refresh API
   - Or skip: many pipelines don't need programmatic BI refresh
4. Compose into one DAG
   Show a complete DAG combining the operator's chosen patterns
5. Trigger it
   Show what the operator sees in the Airflow UI when the DAG runs
```

Each pattern is opt-in. An operator using Python ingestion + dbt + no
BI tool composes patterns B + dbt only. An operator using Airbyte +
dbt + Metabase composes patterns A + dbt + Metabase refresh. The
platform doesn't push them toward any particular stack.

- CeleryExecutor (LocalExecutor stays — matches the cold-tier model)

- A shipped example dbt project
- Recorded tutorial video

## Maintainer-only testing artefacts (in repo, NOT in bundle)

These exist for us to verify v0.1.0 actually works end-to-end. They
live in `system/dags/` and `system/dbt/` but are NOT copied into the
runtime tarball by `release.yml`:

| Path | Purpose |
|---|---|
| `system/dbt/<demo project>` | 3 staging + 2 marts + tests; exercises Cosmos's per-model task generation |
| `system/dags/01_airbyte_dbt_metabase.py` | Verifies Pattern A (HttpOperator-triggered Airbyte → dbt+Cosmos → Metabase refresh) |
| `system/dags/02_python_dbt.py` | Verifies Pattern B (PythonVirtualenvOperator with `requests` → dbt+Cosmos) |
| `system/dags/03_dbt_tests_only.py` | Verifies that operators can run dbt-test schedules independently |

## Verification before tagging

1. Build the orchestack-airflow image locally via a new `make image-airflow` target
2. Run the full stack via `make dev-up`; confirm all 5 images are healthy
3. Place the demo dbt project in the shared volume
4. Trigger DAG `01_airbyte_dbt_metabase.py`; verify in the Airflow UI:
   - HttpOperator successfully triggers Airbyte's API
   - Sensor waits for the sync to complete
   - Cosmos generates one Airflow task per dbt model
   - Models run in dependency order
   - dbt tests run as separate tasks after their models
   - Metabase refresh task succeeds at the end
5. Force a failure mid-dbt by introducing a bad SQL model; verify the
   failure attaches to the right per-model task ID, not the whole DAG
6. Re-run only the failed model from the Airflow UI; verify only it
   and downstream re-execute
7. Trigger DAG `02_python_dbt.py`; verify the PythonVirtualenvOperator
   installs `requests` in an isolated venv on first run and reuses it
   on subsequent runs
8. Confirm orchestrator's audit log records the DAG triggers and the
   warehouse-connection auto-creation
9. Run the PR-gate CI workflow on a draft PR; confirm the 5-image
   matrix all builds + healthchecks

If any step fails, fix and re-verify before tagging.

## Open questions for maintainer sign-off

After the documentation-vs-DAG pivot, the open question list collapses
to two items:

1. **dbt project mount location** — the design uses a single shared
   docker volume `orchestack-dbt-project`. Alternative: per-DAG git
   clone into a temporary directory at DAG run time. Recommendation:
   shared volume — predictable path, one source of truth, no network
   dependency at DAG run time.

2. **Airflow `orchestack_warehouse` connection bootstrap** — orchestrator
   post-start hook auto-creates it. Alternative: walk the operator
   through creating it via the Airflow UI in the first-pipeline doc.
   Recommendation: auto-create — operators shouldn't have to learn
   about Airflow Connections to run dbt.

## Implementation order

1. **Image** — build orchestack-airflow Dockerfile, verify locally
2. **Compose wiring** — update airflow.yml + dbt.yml; add the shared
   volume to docker-compose.yml; update `.env.example` if needed
3. **Orchestrator post-start hook** — auto-create
   `orchestack_warehouse` Airflow Connection
4. **Maintainer testing artefacts** — demo dbt project + 3 verification
   DAGs in `system/dbt/` and `system/dags/`
5. **Documentation** — first-pipeline.html, airflow.html, dbt.html
6. **CI matrix entry** — add 5th entry to ci.yml + release.yml
7. **CHANGELOG + README + _generate_docs.py** updates
8. **Verification checklist** — run end-to-end before tagging
9. **Tag v0.1.0** — push, CI fires, images publish, bundle attaches to
   the release

Once tagged, Docker Hub gets the 5 images at `:0.1.0` + `:latest`, the
GitHub Release attaches the runtime tarball, and the docs site reflects
the working capability.
